Project 4 LockType Sanity Check (Optional)

Q1 Compatibility

2 Points

compatible(A, B) checks if lock type A is compatible with lock type B -- can one transaction have lock A while another transaction has lock B on the same resource?

Q1.1 X Lock Compatibility

1 Point

compatible(X, IS)

compatible(X, IX)

compatible(X, SIX)

Explanation

X locks are incompatible with all other locks, except NL.

// X compatibility sanity checks
assertFalse(LockType.compatible(LockType.X, LockType.IS));
assertFalse(LockType.compatible(LockType.X, LockType.IX));
assertFalse(LockType.compatible(LockType.X, LockType.SIX));
assertFalse(LockType.compatible(LockType.IS, LockType.X));
assertFalse(LockType.compatible(LockType.IX, LockType.X));
assertFalse(LockType.compatible(LockType.SIX, LockType.X));

Q1.2 SIX Lock Compatibility

1 Point

Reminder: SIX locks allow a transaction T1 to read a resource R and all descendants of R, and to request X and IX locks on children of R.

compatible(SIX, IS)

Explanation

If another transaction T2 has the intent to read a child of R this is fine, since T2 might intend to read a child that isn't being modified.

compatible(SIX, IX)

Explanation

If another transaction T2 has the intent to modify a child of R this is not fine, since T1 can still read all children of R.

compatible(SIX, SIX)

Explanation

If another transaction T2 wants to acquire SIX on R this is not fine, since the new SIX lock would allow T2 to modify children of R while T1 can still read those children

// IS compatibility sanity checks
assertTrue(LockType.compatible(LockType.IS, LockType.SIX));
assertTrue(LockType.compatible(LockType.SIX, LockType.IS));
assertFalse(LockType.compatible(LockType.IX, LockType.SIX));
assertFalse(LockType.compatible(LockType.SIX, LockType.IX));
assertFalse(LockType.compatible(LockType.SIX, LockType.SIX));

Q2 Can Be Parent

3 Points

canBeParentLock(A, B) returns true if having A on a resource lets a transaction acquire a lock of type B on a child.

Q2.1 SIX Parent

1 Point

Reminder: SIX locks allow a transaction T1 to read a resource R and all descendants of R, and to request X and IX locks on children of R.

canBeParent(SIX, S)

Explanation

Since holding an SIX lock already allows T1 to read all of R's descendants, requesting an S lock on a child resource would be redundant.

canBeParent(SIX, X)

Explanation

SIX locks allow requesting X locks on children.

canBeParent(SIX, IS)

Explanation

Since holding an SIX lock already allows T1 to read all of R's descendants, requesting an IS lock on a child of R would be redundant.

canBeParent(SIX, IX)

Explanation

SIX locks allow requesting IX locks on children.

canBeParent(SIX, SIX)

Explanation

Since holding an SIX lock already allows T1 to read all of R's descendants and request IX/X locks on children, requesting an SIX lock on a child would be redundant (if the transaction wants to request an X or IX lock on a grandchild it can acquire an IX lock on the parent of the grandchild instead)

// SIX canBeParent sanity checks
assertFalse(LockType.canBeParentLock(LockType.SIX, LockType.S));
assertTrue(LockType.canBeParentLock(LockType.SIX, LockType.X));
assertFalse(LockType.canBeParentLock(LockType.SIX, LockType.IS));
assertTrue(LockType.canBeParentLock(LockType.SIX, LockType.IX));
assertFalse(LockType.canBeParentLock(LockType.SIX, LockType.SIX));

canBeParent(X, X)

Explanation

T1 can already read and write all descendants of R, so requesting an IS lock on a child would be redundant.

canBeParent(X, IS)

Explanation

T1 can already read all descendants of R, so requesting an S lock on a child would be redundant.

canBeParent(X, IX)

Explanation

T1 can already read and write all descendants of R, so requesting an IX lock on a child would be redundant.

canBeParent(X, SIX)

Explanation

// X canBeParent sanity checks

Q2.3 S Parent

1 Point

Reminder: S locks grant a transaction T1 the ability to read a resource R, and read all descendants of R.

canBeParent(S, S)

Explanation

T1 can already read all descendants of R, so requesting an S lock on a child would be redundant

canBeParent(S, X)

Explanation

S does not grant T1 permission to request an X lock on children of R (IX or SIX needed)

canBeParent(S, IS)

Explanation

T1 can already read all descendants of R, so requesting an IS lock on a child would be redundant

canBeParent(S, IX)

Explanation

S does not grant T1 permission to request an IX lock on children of R (IX or SIX needed)

canBeParent(S, SIX)

Explanation

S does not grant T1 permission to request an SIX lock on children of R (IX needed)

// S canBeParent sanity checks
for (LockType lockType : LockType.values()) {
   if (lockType != LockType.NL) {
      assertFalse(LockType.canBeParentLock(LockType.S, lockType));

Q3 Substitutability

4 Points

Function format is substitutable(substitute, required). Checks if one lock type (substitute) can be used in place of another (required). This is only the case if a transaction having substitute can do everything that a transaction having required can do.

Q3.1 NL Substitutability

1 Point

substitutable(S, NL)

substitutable(X, NL)

substitutable(IS, NL)

substitutable(IX, NL)

substitutable(SIX, NL)

Explanation

If a transaction has no lock on a resource then it can't do anything with that resource or children of that resource. Having any other lock type would expand the set of things the transaction can do, so any other lock type can substitute NL

// NL substitutability sanity checks
assertTrue(LockType.substitutable(LockType.S, LockType.NL));
assertTrue(LockType.substitutable(LockType.X, LockType.NL));
assertTrue(LockType.substitutable(LockType.IS, LockType.NL));
assertTrue(LockType.substitutable(LockType.IX, LockType.NL));
assertTrue(LockType.substitutable(LockType.SIX, LockType.NL));

Q3.2 SIX Substitutability

1 Point

Reminder: Having an SIX lock allows a transaction T1 to read a resource R, read descendants of R, request IX locks on children of R, and request X locks on children of R.

(function format is substitutable(substitute, required))

substitutable(SIX, SIX)

Explanation

Substituting SIX with SIX won't change the things T1 can do, so SIX is a valid substitute.

substitutable(S, SIX)

Explanation

Substituting SIX with S would take away T1's ability to request IX and X locks on children, so S is not a valid substitute.

substitutable(X, SIX)

Explanation

Substituting X with SIX would take away T1's ability to request IX and X locks on children. Although this might seem okay, substituting an SIX while it still has IX and X locks on children would lead to redundant IX and X locks. So, X is not a valid substitute.

substitutable(IS, SIX)

Explanation

Substituting SIX with IS would take away T1's ability to read R, read children of R, and to request IX and X locks on children, so IS is not a valid substitute.

substitutable(IX, SIX)

Explanation

Substituting SIX with IX would take away T1's ability to read R and read children of R, so IX is not a valid substitute.

// SIX substitutability sanity checks
assertTrue(LockType.substitutable(LockType.SIX, LockType.SIX));
assertFalse(LockType.substitutable(LockType.S, LockType.SIX));
assertFalse(LockType.substitutable(LockType.X, LockType.SIX));
assertFalse(LockType.substitutable(LockType.IS, LockType.SIX));
assertFalse(LockType.substitutable(LockType.IX, LockType.SIX));

Q3.3 IS Substitutability

1 Point

Reminder: Having an IS lock on resource R allows a transaction T1 to request S and IS locks on children of R

(function format is substitutable(substitute, required))

substitutable(IS, IS)

Explanation

Substituting IS with IS won't change the things T1 can do, so IS is a valid substitute.

substitutable(S, IS)

Explanation

Substituting IS with S would take away T1's ability to request IS and S locks on children of R. Although this might seem okay, substituting an IS while it still has IS and S locks on children would lead to redundant IS and S locks. So, S is not a valid substitute.

substitutable(X, IS)

Explanation

Substituting IS with X would take away T1's ability to request IS and S locks on children of R. Although this might seem okay, substituting an IS while it still has IS and S locks on children would lead to redundant IS and S locks. So, X is not a valid substitute.

substitutable(IX, IS)

Explanation

Since an IX lock on R would allow T1 to acquire any lock type on children of R, T1 can still do everything it could do before with the IS lock. So, IX is a valid substitute for IS.

substitutable(SIX, IS)

Explanation

Substituting IS with SIX would take away T1's ability request S, or IS locks on children of R. Although this might seem okay, substituting an IS while it still has IS, S children would lead to redundant locks. So, SIX is not a valid substitute.

// IS substitutability sanity checks
assertTrue(LockType.substitutable(LockType.IX, LockType.IS));
assertTrue(LockType.substitutable(LockType.IS, LockType.IS));
assertFalse(LockType.substitutable(LockType.S, LockType.IS));
assertFalse(LockType.substitutable(LockType.X, LockType.IS));
assertFalse(LockType.substitutable(LockType.SIX, LockType.IS));

Q3.4 IX Substitutability

1 Point

Reminder: Having an IX lock on resource R allows a transaction T1 to request any lock type on children of R.

(function format is substitutable(substitute, required))

substitutable(IX, IX)

Explanation

Substituting IX with IX won't change the things T1 can do, so IX is a valid substitute.

substitutable(S, IX)

Explanation

Substituting IX with S would take away T1's ability to request any lock type on children of R, so S is not a valid substitute.

substitutable(X, IX)

Explanation

Substituting IX with SIX would take away T1's ability to request any lock type on children. Although this might seem okay, substituting an IX while it still has IS, S, IX, X or SIX locks on children would lead to redundant locks. So, X is not a valid substitute.

substitutable(IS, IX)

Explanation

Substituting IX with IS would take away T1's ability to to request IX, X, and SIX locks on children, so IS is not a valid substitute.

substitutable(SIX, IX)

Explanation

Substituting IX with SIX would take away T1's ability request S, IS, or SIX locks on children of R. Although this might seem okay, substituting an IX while it still has IS, S, or SIX children would lead to redundant locks. So, SIX is not a valid substitute.

// IX substitutability sanity checks
assertTrue(LockType.substitutable(LockType.IX, LockType.IX));
assertFalse(LockType.substitutable(LockType.S, LockType.IX));
assertFalse(LockType.substitutable(LockType.X, LockType.IX));

assertFalse(LockType.substitutable(LockType.IS, LockType.IX));
